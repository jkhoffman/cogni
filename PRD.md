Project Requirements Document: Rust LLM Orchestration Crate v1.0

Overview

Rust lacks a unified framework for building applications with Large Language Models (LLMs) that rivals the flexibility of popular Python libraries. LLM orchestration frameworks (e.g. LangChain) provide standard interfaces for prompts, chains, agents, and memory, making it easier to create AI-driven applications ￼. The Rust LLM Orchestration Crate v1.0 aims to fill this gap by offering a cohesive, type-safe library for Rust developers to compose AI agents and workflows. It will enable developers to integrate multiple LLM providers and tools under one ergonomic API. By leveraging Rust’s performance and reliability, this crate will allow building complex AI agent applications with compile-time safety and efficient concurrency.

In essence, Rust LLM Orchestration will allow developers to define prompt templates, chain LLM calls, incorporate tool usage (such as web search or code execution), and manage conversational memory – all in a model-agnostic way. The library will build on existing low-level Rust crates (for example, using an OpenAI API client crate and Hugging Face’s Candle library for local models) rather than reinventing those wheels. The focus is on high-level abstractions that ensure common errors (like mismatched prompt outputs or tool inputs) are caught at compile time, and that multiple tasks can run in parallel to fully exploit modern multi-core systems. This PRD outlines the goals, requirements, and design for version 1.0 of the crate.

Goals
	•	Unified AI Agent Framework: Deliver a single Rust crate that unifies LLM interactions, tool integrations, and memory into one framework. This makes it easier for Rust developers to build AI agents without stitching together disparate libraries.
	•	Ergonomic and Developer-Friendly API: Provide high-level abstractions (possibly including Rust macros or a DSL) that simplify common tasks like prompt creation and chaining calls. The API should feel natural to Rust developers, emphasizing clarity and ease of use.
	•	Strong Compile-Time Safety: Leverage Rust’s type system so that many errors (like chaining incompatible outputs/inputs between tools and prompts) are prevented at compile-time. The goal is to minimize runtime surprises by catching logic bugs early via types.
	•	Model-Agnostic Design: Support multiple LLM providers through a common interface so developers can switch out models with minimal code changes. v1.0 should work with popular cloud APIs (OpenAI, Anthropic, xAI’s Grok, etc.) as well as local models (via Candle, Ollama).
	•	Tool and Memory Abstractions: Enable agent functionalities beyond simple prompting – including the use of external tools (via a plugin mechanism) and managing stateful memory (conversation history, intermediate results). These capabilities should be built-in and easy to extend.
	•	High Performance & Concurrency: Exploit Rust’s “fearless concurrency” and lack of GIL to allow parallel execution of independent tasks ￼. Agents should be able to perform multiple tool calls or LLM queries concurrently when the task allows, maximizing throughput and efficiency.
	•	Production-Ready Reliability: Ensure the framework is robust enough for real-world use. This means thorough documentation, testing, and sensible defaults for error handling, making it suitable for both prototyping and production scenarios.

Target Audience

The primary audience is Rust developers building AI agents and applications. This includes:
	•	Backend Developers in Rust who want to incorporate AI features (like chatbots, assistants, or autonomous agents) into their systems without switching to Python. They may be familiar with Rust’s ecosystem and seek a library to integrate LLM calls, similar to how Python developers use LangChain or other AI frameworks.
	•	AI/ML Developers who prefer or require Rust for its performance and safety. They might be prototyping complex agents that use multiple tools (web search, code execution, etc.) and need a reliable, type-safe environment for experimentation and deployment.
	•	Enterprise Developers who need to build AI-driven features in a microservice or production environment where Rust is used (for performance, correctness, or integration reasons). The crate should offer the confidence of compile-time checks and efficient concurrency in such settings.
	•	Existing Rust ML Community – developers already using low-level Rust ML or LLM libraries (like Candle, llm-chain, or various API bindings) who want a higher-level orchestration framework. The crate will cater to them by building atop those lower-level libraries, adding the “glue” to compose complex interactions more easily.

In short, any developer who wants to build an AI agent or multi-step LLM application in Rust – from hobbyists to professionals – is the intended user. They should find this crate reduces the boilerplate and cognitive load of orchestrating prompts, tools, and memory manually, while still aligning with Rust’s principles of safety and performance.

Functional Requirements

The Rust LLM Orchestration Crate v1.0 shall provide the following key capabilities:
	•	Multi-Model Integration: The framework must support multiple LLM providers through a unified interface. At minimum, v1.0 will integrate with:
	•	OpenAI (ChatGPT models, etc.)
	•	Anthropic (Claude models)
	•	xAI Grok (the Grok LLM from xAI, Elon Musk’s initiative)
	•	Ollama (for local Llama-family models or others via an Ollama server)
	•	OpenRouter (an LLM router service aggregating many providers)
	•	Requesty (another unified LLM routing platform)
This means developers can use any of these by configuring the client, without code changes to their agent logic. The crate should define a model-agnostic LLM trait or interface that all these providers implement. New providers should be easy to add by implementing this trait. For example, OpenRouter and Requesty aim to streamline access to various LLMs via one API key ￼ – the framework will support them out-of-the-box, and similarly accommodate future services. Local model support will be enabled via Candle (a minimalist Rust ML framework by Hugging Face, used for running models in-process ￼) and via Ollama’s REST API (Ollama provides a simple local LLM server with a REST interface ￼). All model calls (whether cloud API or local) should conform to a common call interface (e.g. a generate() function returning a response type) to allow seamless swapping.
	•	Prompt Templates and Chaining: Provide high-level abstractions for defining prompts and chaining multiple LLM or tool calls into workflows. Developers should be able to:
	•	Define prompt templates with placeholders for variables (e.g. using a mini-DSL or format strings). These templates can be reused and filled with runtime data, enabling dynamic prompt construction for different contexts.
	•	Create chains of calls, where the output of one step feeds into the next. The crate should allow sequential chaining (first call LLM with prompt A, take result, then feed into prompt B, etc.), and possibly branching or merging flows in the future. At v1.0, sequential chaining is the focus, but design should not preclude more complex workflows. Each step in a chain could be an LLM call or a tool invocation, and the framework will handle passing the outputs appropriately.
	•	Ensure type-safe chaining: if one step produces data (text or a structured result) that the next step expects, the types or interfaces should reflect that. For example, a step that calls a math solver tool returning a number should feed into a prompt expecting a number, not raw text. By using Rust types/generics, the library will prevent obvious mismatches at compile time.
	•	Possibly include convenience macros or builders for chaining. (e.g., a macro to write a chain in a linear fashion). This increases ergonomics so that defining a multi-step interaction is straightforward.
	•	Tool Plugin System: Enable LLM agents to use external tools via a standardized plugin interface. The crate should allow registering tools that an agent can call during its reasoning process. In v1.0, the highest priority tools to support (with built-in implementations) are:
	1.	MCP Servers: Support the Model Context Protocol (MCP), an open protocol for connecting LLMs to tools. By integrating an MCP client, the agent can interface with any MCP-compatible tool server. (MCP essentially acts like a “USB-C port” for AI, standardizing how models connect to various tools ￼.) The crate will let developers specify an MCP endpoint, so the agent can call any function exposed by that MCP server (e.g. file system access, code execution, etc. as provided by the server). This opens the door to a wide range of functionality without needing native support for each tool.
	2.	Web Search: A tool for internet search, enabling an agent to answer questions by fetching information. Initially, this will be implemented via an API like SerpAPI or similar, which provides search results in JSON. The framework will provide a SearchTool plugin where the agent supplies a query and the tool returns relevant snippets or links. This allows agents to have up-to-date information retrieval capabilities (e.g. answering questions about current events by searching the web).
	3.	Code Interpreter: A tool that can execute code (likely in a sandboxed environment) and return the results. This could be, for example, a Python interpreter plugin where the agent sends a Python snippet and gets back its output. This is crucial for agents that need to perform calculations, data formatting, or more complex reasoning than the LLM can handle reliably. (Implementation wise, the crate might spawn a subprocess or use a WebAssembly sandbox to run code safely).
	4.	Math Solver: A specialized tool for solving math problems or evaluating mathematical expressions exactly. While an LLM can do math in principle, having a dedicated calculator or algebra system will improve accuracy for numeric tasks. The math tool could parse and evaluate expressions, or use an external library or service (like SymPy or a Rust math crate) under the hood. The agent would use this tool to offload arithmetic or equation solving and then continue the conversation with accurate results.
The plugin system should be designed so that each tool implements a common Tool trait (for example, with methods like fn name(&self) -> &str and an fn invoke(&self, input: &str) -> ToolOutput). Tools can declare what input format they expect and what output they produce. The agent orchestration will include a mechanism for the LLM to decide when to use a tool (this could be via a special prompting technique or function-calling interface with the model). Initially, we might not implement a full autonomous planner in the LLM, but we will allow the developer to manually chain a tool call between LLM calls. Future iterations could allow the LLM to choose tools itself. In any case, the framework must make it straightforward to add new tools as plugins. For example, if a developer wants to add a custom tool (like querying an internal database), they should be able to implement the trait for their tool and register it with the agent.
	•	Memory Management: Incorporate abstractions for “memory” so that agents can maintain state across interactions. Memory here refers to things like conversation history in a chatbot, or storing results from previous chain steps for later use. The crate will provide a Memory interface with multiple backends, including:
	•	In-Memory: A default volatile memory store (e.g. just kept in a Rust data structure) for conversation history or chain state. This is simple and fast, but not persistent.
	•	SQLite: A lightweight file-based database option. The library can use SQLite (via Rusqlite crate, for example) to store chat transcripts or key-value pairs on disk. This allows an agent’s memory to persist between runs without a heavier setup.
	•	Redis: Support using Redis for memory, which is useful in distributed scenarios or when we want a fast, ephemeral store accessible by multiple processes (or even multiple agent instances). The crate would integrate a Redis client to read/write memory entries (with a configurable key namespace).
	•	PostgreSQL: Support a Postgres backend for memory, suitable for robust persistent storage in a production environment. This might be used for logging conversations or sharing memory between services. We can use an async Postgres client or ORM (like SQLx or Diesel) to support this.
The memory interface should be flexible: it might allow storing arbitrary key/value data, or specifically structured conversation data (e.g. a list of messages). For v1.0, a primary use case is conversation buffer memory for chatbots (storing past user and assistant messages so that the LLM can have context). The framework will include this out-of-the-box. Additionally, it should allow intermediate results of chains to be cached or logged. For example, if an agent fetched some data from a tool, memory could store that data for reuse in later steps or for traceability. The system should also make it easy to swap memory implementations (via configuration or feature flags) – e.g., start with in-memory for simplicity, then move to Redis in production by changing initialization. All memory backends must implement a common trait (for example, MemoryStore with methods like save(key, value) and load(key) -> value). This trait will abstract whether the storage is local or remote.
	•	Ergonomic & Type-Safe API: Emphasize a developer experience that is idiomatic to Rust and minimizes runtime errors. Several sub-requirements support this:
	•	The API should be cohesive and easy to learn. Key concepts (LLM model, Prompt, Chain, Tool, Memory, Agent) should be clearly defined in the library with intuitive relationships. Developers should not need to micromanage low-level details like HTTP requests or JSON parsing for each provider – that will be handled internally by integration modules.
	•	Use Rust’s type system aggressively to prevent errors. For instance, each tool could define an input and output type, and the chain composition could be a compile-time checked sequence of transformations. If a developer tries to feed the output of a tool into an incompatible prompt input, it should be a type mismatch caught by the compiler. This strong typing guards against logical errors that would only surface at runtime in Python. (Rust’s powerful static typing can prevent many common programming errors at compile time ￼.)
	•	Provide helper macros or DSLs where appropriate to reduce boilerplate. For example, a macro to easily create prompt templates could ensure at compile-time that all placeholders are filled. We might include macros like prompt!() to define a prompt template, or even chain!() to construct a chain of steps in a more declarative way. These macros would expand to proper Rust code but give the feeling of a domain-specific language for AI agent logic. The design will draw inspiration from crates like llm-chain which use macros for prompts and chain definitions ￼ ￼. The end result should let developers focus on the high-level logic (what the agent should do) rather than glue code.
	•	Ensure extensibility of the API. For instance, if new types of tools or memory are introduced, or new providers, the existing abstractions should accommodate them (via traits, generics, etc.) without breaking changes. Also, the API should allow power users to drop down to lower-level calls if needed (for debugging or custom behavior), while the common path remains simple.
	•	Comprehensive documentation and examples are part of the API’s ergonomics. (Detailed in Documentation Plan, but essentially, the crate’s API will have thorough Rustdoc comments and examples to guide users.)
	•	Parallel and Asynchronous Execution: The framework shall utilize Rust’s concurrency features to allow parallel execution of independent tasks within an agent’s operation. Concretely:
	•	If a chain has multiple branches or steps that do not depend on each other, the crate should allow them to be executed concurrently (e.g. using tokio::spawn for async tasks or Rayon for parallel iterators). One example is a map-reduce operation for summarization: splitting a document into parts, summarizing each in parallel with separate LLM calls, then combining them. Even though map-reduce chains are not explicitly listed in requirements, the design will naturally support such patterns through concurrency primitives.
	•	The plugin/tool system should allow making multiple tool calls in parallel if an agent logic permits. For instance, an agent might decide it needs results from two different tools (say, call a web search and a database query simultaneously) before formulating an answer. The orchestration layer can run those tool futures concurrently and await both results, improving responsiveness.
	•	Likewise, the crate could allow batching or parallel LLM queries when appropriate. For example, if using OpenAI API, maybe send parallel requests for different prompts if they are independent. The design will consider using asynchronous IO throughout, so that waiting on a network call to one API doesn’t block the entire agent. Rust’s lack of a GIL means true multithreading is available for CPU-bound tasks as well ￼. If the agent runs a local model (via Candle) and also a tool that’s CPU-heavy, these could run on different threads in parallel.
	•	All concurrency should be managed in a safe manner, avoiding data races (Rust’s guarantees will help here) and providing cancellation or timeout controls. For example, if one branch of a parallel task hangs or takes too long, the agent framework might have a way to cancel or timeout that call so it doesn’t stall the whole chain.
	•	It’s important that from the developer’s perspective, using these features is straightforward – e.g., the chain API might automatically parallelize certain patterns (like a Chain::parallel([...]) constructor for running multiple sub-chains in parallel, or simple guidance in documentation to use tokio::join!). The complexity of thread/async management will largely be encapsulated by the library.

Summarily, the functional requirements ensure that v1.0 of the crate can handle core LLM orchestration needs: connecting to a variety of models, composing calls and tools, maintaining state, and doing so with a friendly yet safe API. These features will allow building agents such as chatbots with memory, research assistants that call search APIs, or autonomous task solvers that use code execution – directly addressing the common use cases described next.

Non-Functional Requirements

In addition to the core features, several quality and non-functional aspects are crucial:
	•	Type Safety and Reliability: The design must adhere to Rust’s philosophy of catching errors early. All public APIs should be memory-safe (no unsafe needed in user code) and encourage handling of errors via Result types rather than panics. The strong static typing should make it difficult to misuse the library – for example, using the wrong tool output type in a prompt should be a compile error. By preventing common mistakes, we ensure agents built on this framework are more reliable in production. Moreover, the library should be tested with unit tests and integration tests (especially for the chaining logic and tool integrations) to guarantee stability.
	•	Performance and Efficiency: While providing high-level abstractions, the crate should add minimal overhead. Each LLM API call or tool invocation already has inherent latency (network or compute), so the framework should not introduce significant additional latency or CPU cost. Using Rust’s zero-cost abstractions, we aim for performance close to hand-written integration code. Memory usage should be reasonable, and features like parallel execution should scale well on multi-core systems. The crate will take advantage of Rust’s efficient concurrency to outperform similar Python frameworks, especially under load (since Python’s GIL can impede true parallelism ￼). If possible, allow using async runtime (Tokio) for IO-bound tasks and threads for CPU tasks (like local inference or heavy computations) to maximize throughput. Also, consider streaming outputs: for example, if an LLM stream partial results, the framework should allow forwarding those in real-time to the application (this might be in a later version, but the architecture shouldn’t preclude it).
	•	Extensibility and Modularity: The architecture should be modular, making it easy to extend in future versions or by community contributions. This means:
	•	New LLM providers can be added by implementing the common trait (for instance, if Cohere or Google’s PaLM API integration is desired later, or an open-source model runner, it can be done without changing the core).
	•	New Tools can be plugged in similarly by implementing the tool trait. The plugin system might allow external crates to provide additional tools that integrate seamlessly.
	•	The memory interface should handle new backends (e.g., adding a vector database memory in future).
	•	Use Rust’s feature flags to make integrations optional. For example, enable a feature flag for “openai_support” to include the OpenAI API client dependency, or “postgres_memory” for Postgres support. This way, the binary including the crate can slim down by excluding unused integrations. The core should remain lightweight if someone only needs a subset of functionality.
	•	The code should be written in a clean, idiomatic way to invite open-source contributions. Clear trait definitions and separation of concerns will make it easier for others to extend the framework. (For instance, one module for each provider integration, one for each tool, etc., all orchestrated by the core.)
	•	Security and Safety Considerations: As the framework involves executing model instructions and possibly running code, security is important:
	•	The code interpreter tool must be sandboxed (for example, running in a restricted environment so it cannot harm the host system). This might mean using a container, a Seccomp-sandboxed process, or WebAssembly. By v1.0, a simple sandbox approach (like a separate process with limited permissions) should be in place.
	•	When using tools like web search, the framework should guard against prompt injection or malicious instructions. Some basic validation or escaping might be needed (e.g., if an agent prints out web content, ensure no HTML/script injection in output). This might be more of an application concern, but the framework can provide guidelines or helper to sanitize tool outputs if needed.
	•	API keys for services (OpenAI, etc.) will be needed; the framework should not expose these or log them. Provide a secure way to configure credentials (like through environment variables or config files that the crate can read, or requiring explicit passing of keys into client objects). We should document best practices to keep keys safe.
	•	The crate’s own code should of course avoid unsafe memory handling and follow Rust’s safety guarantees.
	•	Documentation and Usability: High-quality documentation is a must (expanded in the Documentation Plan section). The project should adhere to the Diátaxis documentation framework, producing Tutorials, How-To Guides, Reference docs, and Explanation material ￼. This is a non-functional requirement because the success of the library depends on developers being able to understand and use it effectively. Each public API item should have documentation comments (to appear on docs.rs or a documentation site), including examples of usage. The repository should include the example use cases as fully working programs that users can study and run. Additionally, the project should have a quick-start guide such that a developer can get a simple agent running (for example, a hello-world chatbot) in minutes. Good documentation and examples will lower the learning curve and improve adoption.
	•	Compliance and Licensing: The crate will be open-source (likely dual-licensed MIT/Apache-2.0, which is common for Rust libraries) so that it can be used freely in both open-source and proprietary projects. This encourages community contributions and commercial adoption. We should ensure any third-party crates we rely on are compatible with this licensing. Also, any code or prompts we include should not violate the usage policies of the LLM providers (for instance, using the OpenAI API in accordance with their terms). While not a direct library requirement, we should be mindful of compliance with terms of service for integrated services.
	•	Testing and Quality Assurance: As a non-functional goal, the project should maintain high code quality. Continuous integration tests should cover:
	•	Unit tests for prompt template interpolation, chain execution logic (with maybe a mock LLM to simulate responses), and memory stores.
	•	Integration tests hitting actual APIs (maybe conditionally, or using recorded responses) to ensure each provider integration works (OpenAI, Anthropic, etc.).
	•	Example scenarios (the use cases described) should be tested to ensure they produce expected outcomes.
This ensures that v1.0 is robust on release. Also, performance tests or benchmarks might be included to verify the overhead remains low.

In summary, non-functional requirements ensure the crate is safe, fast, extensible, and well-documented. Rust’s core strengths (safety, speed, concurrency) should shine through in the implementation, making the framework not just feature-rich but also trustworthy for developers to build upon.

Example Use Cases

To illustrate the capabilities of the Rust LLM Orchestration Crate v1.0, here are a few common use cases and how the framework addresses them. Fully working examples for each of these will be included with the project (serving as both validation and guides for users).

1. Chatbot with Memory

A developer wants to create a chatbot that can carry on a conversation with a user, remembering context from earlier in the dialogue. For example, a customer support bot that can refer to what the user said two questions ago, or a role-play chatbot that maintains character details throughout a session.

Using the orchestration crate, the developer can define a conversation chain: each user message plus the bot’s reply is stored in a memory buffer. The prompt for the LLM at each turn is generated from a template that includes the recent conversation history (pulled from the memory interface) along with the new user query. The memory management feature (perhaps using SQLite or Redis for persistence between sessions) ensures the history is retained. The crate’s prompt template system makes it easy to format the conversation (e.g., as a series of “User: … / Assistant: …” lines) without manual string manipulation. Strong type-safety ensures that, for instance, the memory returns the right type of objects to insert into the prompt (a list of message structs that can be formatted).

For the LLM itself, the developer could choose OpenAI’s GPT-4 via the OpenAI integration. The code would not have to deal with HTTP calls or JSON parsing of responses – they simply call a method like llm.generate(prompt, history) and get back the assistant’s answer. If later they want to switch to an Anthropic Claude model, they could configure the agent to use the Anthropic client with minimal code changes, thanks to the model-agnostic design. The end result is a chatbot that seamlessly maintains context over dozens of turns, which would be demonstrated in an example (e.g., a console chat application using the crate, showing the bot remembering earlier details from the conversation).

2. Agent that Searches and Synthesizes

In this scenario, an “AI research assistant” agent is tasked with answering a question by searching the web and then synthesizing the information into a coherent answer. For example, a user might ask: “What are the latest developments in renewable energy storage?” The agent should perform a web search for that query, read through the results, and produce a summary answer.

With the orchestration framework, the developer can create an agent chain like:
	1.	Search step: Use the Web Search tool plugin. The agent passes the user’s query to the SearchTool (integrated via SerpAPI or similar) which returns a list of top results or snippets.
	2.	LLM synthesis step: Take the search results and feed them into an LLM prompt that asks the model to summarize or compile an answer using the found information. For instance, the prompt template might be: “You are a research assistant. The user asked: ‘{{question}}’. I have gathered the following information: {{search_results}}. Please provide a concise summary answer.” The placeholders {{question}} and {{search_results}} are filled with the user’s query and the content retrieved by the search tool, respectively.

The framework handles passing the output of the search tool (likely a structured format, e.g. a list of text snippets) into the prompt for the LLM. A robust implementation might limit the number of snippets or the total length to fit model context – the developer can control that in their chain setup, while the framework provides convenient data structures (maybe a SearchResult type). The actual LLM call could be to OpenAI or another provider as configured.

This use case demonstrates tool integration and chain orchestration. The agent effectively augments the LLM with a tool to fetch real-time data. Thanks to Rust’s concurrency, the search step could even be done in parallel with other tasks if needed (though in this simple chain it’s a single path). After retrieving results, the heavy lifting of analyzing text is done by the LLM. The final answer is then returned to the user. The example included might show how just a few lines of code with this crate can perform a web search and get an answer (contrasting with how much boilerplate it would be if done from scratch). It also shows the benefit of a model-agnostic approach: the same logic could be used with different LLMs – if one model is better at summarization, just plug that in.

3. Multistep Planner Calling Tools

This use case involves a more complex agent that can break down a problem into multiple steps and invoke various tools to arrive at a solution. Think of a “task-solving agent” which, given an objective, can plan a series of actions (some actions might be calls to tools, others are reasoning via LLM) to achieve the goal. For example, consider an agent asked: “Analyze this dataset and tell me if there are any interesting trends.” The agent might need to: 1) read a dataset file, 2) run some code to analyze it (like computing statistics or charts), and 3) generate a report of findings.

Using the crate, the developer can implement a simplified planner:
	•	First, the agent uses the LLM to formulate a plan. A prompt could be designed for the LLM like: “You are a planning agent. The user’s request is: {{request}}. Break this task into clear steps and indicate which tool or action to use for each step.” The model might output something like: “Step 1: Use File tool to open the dataset. Step 2: Use Code tool to compute statistics. Step 3: Use LLM to summarize findings.”
	•	The framework can parse this plan (maybe the plan is returned in a structured format or the developer can parse the text). With the plan in hand, the agent orchestrator then executes step by step:
	•	File Tool (if we had such, or maybe via MCP if an MCP server provides file access) to get the dataset content.
	•	Code Interpreter Tool to run analysis code. Perhaps the agent has a predefined analysis script or it could even ask the LLM to generate code to execute (another step: “LLM, please write Python code to analyze dataset for trends.” Then run that code with the interpreter tool).
	•	LLM final step to take results (say the code produced some numbers or a graph description) and formulate a user-friendly answer.
	•	Throughout these steps, memory can be used to store intermediate results: the file content, the code output, etc., making them available to subsequent steps. The memory could be in-memory if it’s all within one run, or persistent if the agent needs to recall something later or if we want to log all steps.

This multi-step sequence tests the full capability of the orchestration crate: it involves dynamic decision making (the plan), multiple different tools, and multiple LLM calls in one workflow. The crate’s role is to provide the backbone to make this happen smoothly:
	•	The prompt template and chain system is used for the planning prompt and for subsequent LLM calls.
	•	The tool plugin system executes the specified actions (file access, code run).
	•	The type-safety ensures, for instance, that the code execution tool returns output in a format that the next prompt can accept (maybe as a string or as structured data).
	•	The concurrency aspect could allow some steps to run in parallel if the plan allows (though in a strict sequence it might not, but if the plan had independent sub-tasks, those could be parallelized).
	•	Error handling is important here: if one step fails (say code tool encountered an error), the framework should catch that and allow the agent to handle it (maybe by prompting the LLM to react to the error). For v1.0, simply propagating an error out is acceptable, but we design with the idea that agents might need to recover from tool failures gracefully.

The example delivered with the project might be a simplified version of the above (to keep it manageable), but it will show an agent using at least two tools and an LLM to solve a task. For instance, an agent that given a math word problem, decides to use the math solver tool for the calculation part, and then uses the LLM to explain the result. This demonstrates multi-step reasoning and tool use. It’s essentially a taste of an autonomous agent – and it showcases how the framework enables such patterns with far less effort than coding each step and integration manually.

These use cases confirm that the functional requirements cover real-world needs. They will serve as validation tests for v1.0: the framework should be able to implement each scenario with relative ease and clarity. The included examples (with code in the repository) will double as tutorials for new users.

Architecture Overview

The architecture of the Rust LLM Orchestration Crate is designed in layered components, balancing high-level abstraction with the integration of low-level capabilities (via existing crates and services). Below is an overview of the main architectural components and how they interact in v1.0:
	•	LLM Integration Layer: At the core lies a trait (e.g., LanguageModel) that defines a common interface for all LLMs. This might include methods like complete(prompt, params) -> LlmOutput or chat(conversation) -> LlmOutput. Each supported provider (OpenAI, Anthropic, Grok, etc.) will have a concrete implementation of this trait. Internally, these implementations will use the provider’s API: for OpenAI, likely an HTTP client (wrapping an existing crate or using reqwest with the OpenAI REST endpoints); for Anthropic, their HTTP API (maybe via a community crate or direct REST calls); for Grok (xAI) if an API is available, similarly through HTTP calls. OpenRouter and Requesty might not need a separate implementation if they mimic OpenAI’s API – possibly the OpenAI client can be pointed to the OpenRouter URL with an API key, since OpenRouter provides a unified OpenAI-compatible API for multiple models ￼. We will verify if their interfaces align and either treat them as special cases or just as configuration of an OpenAI/Anthropic client. Ollama integration will use its REST API (running locally on the user’s machine). The crate might include a small Ollama client module that sends HTTP requests to http://localhost:11434/api/generate with the prompt and model name (as shown in their docs ￼). For local inference via Candle, we’ll integrate Candle’s model execution APIs. For example, if Candle supports generating text from a loaded transformer model, our Candle-based LanguageModel implementation will load a model (perhaps from a file or from Hugging Face) and run it in-process to produce outputs. This might require managing model initialization and ensuring it runs on a separate thread or is async if it’s long-running (so as not to block everything else). The LLM Integration layer thus abstracts whether a call goes out to an API or is computed locally. All LLM outputs can be normalized to a common format (e.g., a struct containing the text and maybe usage stats/tokens, etc.). By structuring it this way, higher layers (chains/agents) don’t worry about the specifics of each provider.
	•	Tools Plugin Interface: The architecture defines a Tool trait to represent external tools/actions an agent can use. For v1.0, we will implement a few built-in tools (Search, Code, Math, plus MCP as a gateway to many tools). The Tool trait might look like: fn invoke(&self, input: &str) -> Result<ToolOutput, ToolError> along with metadata like fn name(&self) -> &'static str and a description (for use in prompts if the LLM needs to choose tools). Each concrete tool (e.g., WebSearchTool, CodeInterpreterTool, MathTool, McpTool) will live in its own module. The WebSearchTool will depend on an HTTP client to call the search API (SerpAPI or others); the developer will need to provide an API key via config. The CodeInterpreterTool will manage a sandboxed process – for instance, it might launch a Python process with restricted permissions. We could use an existing crate for sandboxing or just impose restrictions (like a time limit and memory limit on the code run). The code tool will take as input a code snippet (and perhaps some optional context data) and return the stdout or result of executing that code. The MathTool could be implemented by a simple parser (to evaluate expressions safely) or by calling out to a service – but likely an internal implementation using Rust libraries (e.g., evaluating arithmetic or using meval crate for math expressions). The McpTool is a special one: rather than a single function, it’s a client that can route to many functionalities on an MCP server. The MCP protocol likely uses a message-based interface (the Anthropic description suggests a standard way to call named functions with arguments ￼). We will include an MCP client component that can connect to an MCP server (possibly via stdio or network). It might maintain a connection so that multiple calls can be made. When the agent wants to use a specific MCP-exposed tool, our McpTool will format the request per MCP spec and send it, then wait for the result. The advantage is that by implementing MCP, we automatically support any tool that someone exposes via that protocol (file access, browser control, etc., without writing custom code for each). In the architecture, tools are pluggable: the agent or chain can have a registry of available tools. The agent logic (either automated by LLM or predetermined in the chain) can pick a tool by name and invoke it through this unified interface. This design allows adding new tools later by just adding a struct implementing Tool. The system could even allow dynamic discovery of tools (for example, if an MCP server lists what it has).
	•	Agent/Chain Orchestrator: At a higher level, the crate will likely provide an Agent struct or a set of functions that orchestrate the flow between prompts, LLM calls, and tools. This is where the “chain of calls” logic lives. An Agent could be configured with:
	•	A selected LanguageModel implementation (e.g., an OpenAI client).
	•	A set of Tool implementations (optional, if we want the agent to have tool-using ability).
	•	A MemoryStore for keeping state.
	•	Perhaps a strategy for how to use these (e.g., always follow a fixed sequence vs allow LLM to dynamically decide).
In v1.0, we might implement a simpler form of agent that follows a predetermined chain (scripted by the developer via the chaining API). The agent orchestrator will execute the steps in order:
	1.	Take initial input (question or task).
	2.	Fill in a prompt template and call the LLM for either the final answer or the next action.
	3.	If the next action involves a tool, call the appropriate tool plugin.
	4.	Store any intermediate results in memory as needed.
	5.	Possibly loop or iterate if the design calls for multi-step planning (for example, a simple loop where the LLM output is checked if it indicates needing a tool, then the tool is executed, result fed back to LLM, and so on).
	6.	Eventually produce a final LLM result as output.
The agent orchestrator will heavily use the other components (LLM interface, Tools, Memory) and coordinate them. It will also incorporate concurrency where relevant: for instance, if the agent has to call two different tools to gather information before an LLM step, it could do so in parallel. The orchestrator will likely be implemented using async/await (requiring the user to run it in a Tokio runtime or similar, which is common for Rust async APIs). If users prefer sync, we might offer both, but async is important for I/O concurrency. We will ensure that the design of the orchestrator cleanly separates the definition of the chain from the execution, so that one could even serialize a chain or inspect it for debugging (for example, log each step and tool call). The architecture could also allow a lower-level use: developers could manually drive the sequence (calling the LLM and tools themselves using the primitives) if they need more control, but the agent orchestrator provides the convenient automation of the sequence.
	•	Memory Module: The memory component will be abstracted behind a trait like MemoryStore (for persistent memory) and perhaps a simpler one for transient conversation context (like ChatMemory that could just be an in-memory list of messages). The architecture will have specific adapters:
	•	An in-memory adapter (just holds data in a struct, cleared when agent is dropped).
	•	A SQLite adapter (manages a file DB; perhaps the adapter will handle creating a table if not exists and inserting conversation turns with session ids, etc.).
	•	Redis adapter (requires a Redis connection – could use redis-rs crate – storing data with keys, maybe using a hash or list structure for conversations).
	•	Postgres adapter (using sqlx or diesel to define a schema for memory storage).
The Memory module will expose methods for the orchestrator or agent to save information (like after each interaction, save the user query and LLM answer) and load information (like retrieve the last N messages to include in context). There might be different strategies (e.g., a short-term memory that just fetches recent messages for context window, and possibly a long-term memory that stores all interactions). For v1.0, basic support means the developer can choose one of the backends and the agent will use it to store the conversation or chain state. The architecture needs to allow initialization of these – e.g., an AgentBuilder might accept a memory: Box<dyn MemoryStore> that could be constructed by a helper function (like MemoryStore::postgres(connection_string)). Because database connections or clients might be async, the memory trait might have async methods. We also need to handle errors (failed DB writes, etc.) gracefully within the orchestrator (likely just propagating with context). The memory trait might be generic over what type of data it stores (perhaps using serialization for arbitrary data). However, focusing on conversation text simplifies it for now. The key point is that memory is pluggable – switching from in-memory to Redis is just a config change.
	•	Macro and DSL Support: A noteworthy architectural feature is the use of Rust’s macro system to create a kind of internal DSL for agent definitions. The crate will include declarative macros, which are expanded at compile time to regular Rust structs/function calls. For example, a prompt! macro can take literal text with placeholders and produce a PromptTemplate struct. It might even check that placeholders are valid Rust identifiers and could generate a struct type that requires those fields to be provided (enforcing at compile time that, say, “{{user_name}}” is provided in the context before using the prompt). Similarly, a chain! macro might allow writing something like:

chain! {
    step1: LLM(prompt = "You are ..."), 
    step2: Tool(name = "math_solver"), 
    step3: LLM(prompt = "Explain the result of step2 to the user")
}

and this would expand into the code needed to wire step1’s output into step2, etc. The architecture would parse the macro input and use procedural macro capabilities if needed for more complex validation. Alternatively, a fluent builder API could be provided (which might be easier to implement initially). Regardless, the crate’s design will treat these macros as sugar on top of the core structs. They are not strictly necessary, but they greatly improve developer experience by reducing boilerplate and highlighting errors at compile time if the chain is mis-specified. This approach is analogous to how some web frameworks in Rust provide DSLs (like Diesel for SQL queries with macros) or how llm-chain crate has macros for prompts ￼. In our architecture, the macros will reside in the crate (or a sub-crate if needed to separate for compile times) and will construct instances of the core types (PromptTemplate, Chain, etc.). This maintains a separation of API facade vs core logic.

	•	Concurrency and Execution Engine: Under the hood, we will use Rust’s async runtime (Tokio) to manage asynchronous operations. The orchestrator will be largely async – each LLM call or tool invocation that involves I/O will be .await-ed. We might use tokio::join! to run things in parallel when possible, or spawn tasks. If the crate spawns tasks internally (for parallel tools), it will ensure to collect and handle their results (and propagate errors). We should also consider a cancellation mechanism – if the user drops the agent future (cancelling the operation), all spawned tasks should be stopped to avoid background work. Using Tokio’s cooperative cancellation via dropping futures should suffice. For CPU-bound local tasks (like running a Candle model or executing code), these could be run on separate threads to not block the async runtime’s core. We may utilize tokio::task::spawn_blocking for such cases. The architecture also can take advantage of Rust’s multi-threading beyond async: for example, if two independent LLM calls need to run truly simultaneously (not just interleaved), one could opt to run multiple LanguageModel::generate calls on separate OS threads. Since Rust has no GIL, this yields actual parallelism ￼. We might provide configuration for thread pool sizes if needed (though by default, Tokio’s runtime can handle many concurrent tasks). The execution engine also involves error propagation and retry: If an API call fails (network error, rate limit), the orchestrator could catch that and either retry or return an error. For v1.0, a simple strategy might be one retry for transient errors. More advanced logic (circuit breakers, adaptive retries) could be added later or left to the user to configure. Logging is another aspect: the architecture should log important events (if a logger is provided) such as “Calling OpenAI model X with prompt Y”, “Tool math_solver returned result R”, etc., to aid debugging and observability. We will integrate with the standard Rust log or tracing crates, so that users can enable debug logs to see the chain of events. This doesn’t affect the structure per se, but it’s part of the execution considerations.

Diagram: The Diátaxis documentation framework divides docs into four quadrants, addressing distinct user needs – Tutorials and How-to guides (action-oriented), and Explanation and Reference (information-oriented) ￼. Our documentation plan will follow this structure to ensure comprehensive support for users.

Documentation Plan

Proper documentation is critical for the adoption of the library. We will structure all documentation following the Diátaxis framework, which identifies four key types of docs: Tutorials, How-To Guides, Reference, and Explanation ￼. Each serves a different purpose, as described below. The plan is to provide full documentation in each category by v1.0 release:
	•	Tutorials (Learning-Oriented): These will be beginner-friendly, step-by-step guides that teach users how to achieve something with the crate from scratch. They assume no prior experience with this library (though basic Rust knowledge is expected). We plan to include at least:
	•	Tutorial 1: Getting Started – Building Your First AI Agent. This tutorial will walk through a simple use case, perhaps creating a chatbot that echoes user input with slight modification, just to cover basics. It will show how to add the crate as a dependency, set up an OpenAI API key (for example), and write a minimal main function that uses an Agent to have a single turn conversation. The goal is to get something running quickly to show the library in action.
	•	Tutorial 2: Chatbot with Memory. Using the first example use case, this tutorial will be more involved: building a multi-turn chat assistant that remembers context. It will introduce using a Memory backend (maybe start with in-memory, then show switching to SQLite). The tutorial will step through writing the prompt template for conversation, configuring the agent with memory, and running a sample dialogue.
	•	Tutorial 3: Using Tools (Web Search example). This will cover how to integrate and invoke tools. It could use the “search and synthesize” scenario. The tutorial will likely demonstrate how to register the SerpAPI key, enable the SearchTool, and then construct an agent that uses it in a chain. This teaches the user how to incorporate a tool call between LLM calls. It will show the code and also how to examine the agent’s process (maybe logging each step).
	•	If time permits, a fourth tutorial might cover the multi-step planner example, but that might be more advanced. At minimum, the above three ensure we cover a basic LLM call, memory, and tool usage – the three pillars of functionality.
Each tutorial will be written in a narrative, lesson-like style, focusing on the “why” and “how” at a beginner level. They will include plenty of code snippets and expected outputs. By following the tutorials, a developer should be able to progressively build up to more complex agent scenarios.
	•	How-To Guides (Goal-Oriented Recipes): These are shorter, problem-solution docs for specific tasks a user might want to accomplish with the crate. They assume the user knows the basics (maybe did a tutorial or two) and now needs to know “How do I do X?”. Planned how-to guides include:
	•	How to Add a New LLM Provider. A guide that shows how one would extend or configure the framework to use a provider that might not be built-in. For example, if someone wants to use a new API (say Cohere or a self-hosted model), this guide would walk through implementing the LanguageModel trait for that provider, and how to plug it into the agent. This doubles as an internal developer guide in case contributors want to add more providers.
	•	How to Implement a Custom Tool. This guide will show the steps to create a new tool plugin. For instance, if a developer has an internal API or wants to add a specialized calculator, how can they do that? The guide will cover defining a struct for the tool, implementing the trait (with an example of processing input and producing output), and then registering it so the agent can use it. We might use a simple example like a “DateTimeTool” that returns the current date/time, as the custom tool to illustrate the process.
	•	How to Change Memory Backend. A guide focusing on switching memory storage. Perhaps the user started with in-memory and now wants persistence. This will show how to enable the SQLite feature (e.g., via Cargo features), supply a database path, or how to configure Redis connection for the memory. It will also mention any setup needed (like running a Redis server). Essentially, it ensures users can easily move between memory options.
	•	How to Configure Concurrent Calls or Parallel Chains. This guide may illustrate using the concurrency features. For example, if the user wants to perform two queries at once, how do they do that with the API? It might cover using a parallel chain method or manually spawning tasks with the library’s primitives. This will help users optimize their agents for performance.
	•	How to Use the Agent in a Web Server (integration scenario). A guide showing an example of integrating the agent into a warp or axum web server route, to answer HTTP requests. This is more about context integration but is useful for real applications. It would show that the agent can be long-lived or recreated per request, and how to manage things like one agent per user session (tying into memory).
Each how-to is focused on a single question and provides a concise answer with code snippets. They serve as a reference for users who have a specific goal in mind and need to know the steps to achieve it, without the explanatory overhead of a full tutorial.
	•	Reference (API Reference): This will be the exhaustive technical reference for all public components of the crate. Largely, this will be covered by Rustdoc-generated documentation from source code comments. We will ensure that every public struct, trait, enum, and function has documentation comments explaining its purpose, parameters, and usage. In addition:
	•	We might create a Reference Guide section in documentation that organizes these APIs by concept. For example, a page listing all LLM Provider Integrations with links to their structs/traits, a page for Tools listing each tool and its API, etc. This can be generated by hand or using a documentation generator.
	•	The reference should also include things like error types (so developers know what errors to handle), configuration structures, and any macros provided by the crate (with explanation of their syntax and examples).
	•	Being a v1.0, the API surface might still be evolving, but we’ll treat the reference as something developers can rely on for exact details. For instance, if they want to know the parameters for the OpenAIClient::new() or the fields in ToolOutput, it’s all clearly documented.
	•	We will also list any differences or nuances for each provider if applicable (e.g., “Anthropic’s model may require prompting with a system message in a certain way, here’s how the crate handles it”).
Essentially, the Reference is the authoritative spec of the library’s interface – it’s where someone who is coding can quickly look up function signatures or trait bounds. Following Diátaxis, reference material is dry and factual, not tutorial in style ￼. We’ll ensure it’s complete and accurate.
	•	Explanation (Background & Discussion): These documents will provide the rationale, theory, or internal workings to help users (especially advanced ones) deeply understand the framework. While not everyone will read these, they add transparency and learning opportunities. Planned explanation topics:
	•	Design Philosophy & Architecture. A write-up discussing why the crate is designed the way it is. For example, why we chose traits and generics for abstraction, how compile-time safety guided our decisions, how the layers (LLM, Tool, Memory) interact. This is somewhat like an expanded version of the Architecture Overview in this PRD, but formatted as a part of docs. It helps users understand the mental model of the crate and could also assist contributors or those extending the system.
	•	Understanding Chains and Agents. An explanation of the pattern of LLM chains and agent loops. This might include a conceptual comparison to how it’s done in other frameworks (like explaining what “ReAct” style is, how our framework could implement it, etc.). It provides context on various agent paradigms (reflecting on the state-of-the-art of LLM agents) and how the crate can be used to realize them. This isn’t a how-to, but rather a discussion so users can conceptualize complex uses.
	•	Memory and Context Management. A deep dive into the memory module: why memory is important in LLM applications, the trade-offs of different memory stores (performance vs persistence), how our crate’s memory handling ensures the context is properly injected, and strategies like windowed memory or summary memory (even if not fully implemented in v1.0, we can discuss how one might do it with the framework).
	•	Tool Use and MCP. A narrative explaining the challenge of extending LLMs with tools, how Model Context Protocol works in principle, and how our tool interface aligns with that. It can talk about the security considerations of tools, or how the agent chooses to use a tool (e.g., via prompt engineering, since the LLM has to be told when to use which tool if doing autonomously).
	•	Concurrency in Rust for LLM workflows. An explanation doc that might compare (at a high level) how doing parallel tasks in Rust differs from doing them in Python (mentioning the GIL and why we can achieve true parallelism ￼). This is both educational and builds trust that Rust is a good choice for these workloads. It may also describe how we prevent race conditions (Rust’s ownership means shared state like memory store must be thread-safe or cloned).
These explanations serve to educate the user on the “why” behind certain features. They are important for those who want to become power users of the library or contribute to it.

In producing these documents, we will ensure cross-referencing between them. For instance, a tutorial might link to a reference page for a specific function, a how-to guide might refer to an explanation for deeper background. This interlinking helps users navigate to the depth of information they need ￼. We will likely host the documentation on a web platform (like GitHub Pages or docs.rs for reference and an mkdocs site for guide-like docs) for easy access.

Additionally, the project repository will include an examples directory with well-documented example programs (corresponding to the use cases above). These examples will be kept up-to-date with the code and will be tested (so that we know they work with each release). The examples serve as an immediate resource for those who prefer to learn by code reading rather than reading documentation prose.

Finally, we will maintain a CHANGELOG and upgrade guide (if needed) even in v1.0 to document any breaking changes leading up to 1.0 and set the stage for users to follow along in future versions. Transparency in changes and clear documentation are key to developer trust.

By following the Diátaxis structure, we ensure that every user – whether a novice just trying out the library, or an experienced developer integrating it into a complex system – can find appropriate documentation to help them succeed. Our goal is not only to deliver a powerful library, but also to make sure it’s understood and utilized effectively, through excellent documentation materials.

Conclusion

This PRD for the Rust LLM Orchestration Crate v1.0 outlines a comprehensive plan to create a powerful, developer-friendly framework for building AI agents in Rust. By meeting the functional requirements with a careful architecture and paying attention to non-functional qualities and documentation, the project aims to empower Rust developers in the AI domain. Upon completion of v1.0, developers should be able to confidently build applications that integrate multiple language models, perform complex reasoning across tools, and maintain conversational context – all with the reliability and performance that Rust provides.

With these requirements and plans in place, the next steps would be to validate feasibility (spiking integrations for a couple of providers and tools), then proceed with implementation iteratively, constantly referring back to this PRD to ensure the final product aligns with the vision set out here. The delivered crate will be a significant step toward parity with Python’s AI ecosystems, and in some ways, it will exceed them by leveraging Rust’s strengths (type safety, concurrency ￼, etc.), potentially setting a new standard for what an LLM framework can offer in terms of robustness and speed.
